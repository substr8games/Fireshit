# Fireshit Security Policy
## Supported releases

Security fixes are provided for the current canonical Fireshit release and any additional release explicitly identified as supported.

## Private reporting

Report suspected vulnerabilities through **GitHub Security Advisories → Report a vulnerability** in the canonical `substr8games/Fireshit` repository. If that private form is unavailable, open a public issue containing no exploit details or sensitive data and request a private reporting channel from Giz@SUBSTR8Games.

Do not place an unpatched exploit, private user data, credentials, or destructive reproduction instructions in a public issue.

A useful report includes:

- affected version;
- affected module;
- environment;
- impact;
- reproduction steps;
- proof of concept where safe;
- suggested mitigation, if known; and
- whether the issue has been disclosed elsewhere.

## Good-faith research

Good-faith security research is authorized when the researcher:

- tests only systems and copies they own or are authorized to test;
- avoids accessing unrelated personal data;
- avoids persistence and destructive modification;
- avoids service disruption;
- does not extort, threaten, or demand payment;
- reports the issue privately;
- supplies enough information to reproduce it; and
- allows a reasonable coordinated-disclosure period.

This authorization does not grant permission to violate third-party systems, privacy, contracts, or law.

## Disclosure

Giz@SUBSTR8Games may coordinate a disclosure date based on severity, exploitability, available mitigation, and release readiness.

The project may credit the reporter using the name they request.

No bug bounty or payment is promised unless announced separately in writing.

## Ownership

A security report does not transfer ownership of the reporter’s independent research.

Submission does not create an automatic confidentiality agreement beyond the coordinated-disclosure expectations stated here.
